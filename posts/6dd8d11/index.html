Hace ya aproximadamente un año que comencé mi aprendizaje en el desarrollo de apps Android nativas, utilizando para ello las herramientas y recursos que propone Google como oficiales. Desde entonces han cambiado distintas cosas, desde el sistema de diseño de UI, hasta el modelo de gestión de dependencias. Este último es en el que he percibido más cambios.

Historia y contexto

Para aquellos que no tengan idea sobre el desarrollo Android, la configuración del proyecto se gestiona con Gradle, esto es quizás lo único que no ha cambiado. En el inicio de mi aprendizaje recuerdo que me tope con un build.gradle escrito en Groovy, muy distinto a lo que se puede ver hoy. Si mal no recuerdo se trataba de una app ToDo o una app para agregar frases y para ello utilicé Java en lugar de Kotlin, y era a este punto donde quería llegar.

Desde la aparición de Kotlin se ha ido imponiendo el uso del lenguaje en la mayoría de partes de un proyecto, en primer lugar se sustituyo parcialmente el uso de Java en favor de Kotlin, posteriormente se hizo lo mismo con el sistema de vistas XML y el archivo de configuración del proyecto, ambas cosas sustituidas por Jetpack Compose y Kotlin respectivamente.

Jetpack Compose es un sistema para la creación de vistas que utiliza Kotlin como lenguaje para este propósito.

Nuevo sistema para gestionar dependencias

Lo anterior no es lo que ha motivado que haya escrito este texto. Lo realmente llamativo es que tanto el build.gradle del módulo app como el del modulo de proyecto ya no son los encargados de alojar las dependencias del proyecto.

Esta labor recae ahora en el fichero libs.versions.toml. En un inicio me resultó algo confuso, ya que de la noche a la mañana el aspecto que tenía el build.gradle al crear un proyecto nuevo había cambiado, pero no había cambiado del todo. Es decir, en la base del documento se siguía mostrando la lista de dependencias, pero se mostraban con aspecto más limpio y totalmente distinto a lo que acostumbraba a ver.

Trate de añadir una nueva dependencia, tal y como lo había hecho desde ese entonces hacía detrás, pero al guardar los cambios y actualizar comenzaron a aparecer muchos errores.

Esto se debía a que, como ya mencioné, build.gradle no es el responsable de alojar dependencias del proyecto, por lo tanto, resulta inútil crear un implementation() como se venía haciendo hasta ahora.

libs.versions es un fichero toml con tres bloques bien diferenciados, versions, libraries y por último plugins.

En su estructura encontrarás en primer lugar, versions, como su nombre indica, se utiliza para declarar o establecer las versiones de las dependencias, para posteriormente referenciarlas, bien en libraries o en plugins.

[versions]

Tendría un aspecto similar al siguiente:

[libraries]
androidx-core-ktx = { group = "androidx.core", name = "core-ktx", version.ref = "coreKtx" }
junit = { group = "junit", name = "junit", version.ref = "junit" }
androidx-junit = { group = "androidx.test.ext", name = "junit", version.ref = "junitVersion" }
androidx-espresso-core = { group = "androidx.test.espresso", name = "espresso-core", version.ref = "espressoCore" }
androidx-lifecycle-runtime-ktx = { group = "androidx.lifecycle", name = "lifecycle-runtime-ktx", version.ref = "lifecycleRuntimeKtx" }

Esta sección está definida como libraries y contiene tablas de pares clave-valor, que son, group, name y version.ref.

Las antigua estructura de implementation tenian un aspecto como este:

"androidx.lifecycle:lifecycle-viewmodel:$lifecycle_version"

En ella se ven tres partes diferenciadas por el simbolo : la primera parte corresponde al grupo al que pertenece la librería, en este caso, androidx.lifecycle, el segundo pertenece a la librería que se añadirá al proyecto, lifecycle-viewmodel y la tercera y última parte corresponde a la versión.

Cada tabla de pares se debe definir con el nombre de la librería añadida separando cada palabra por guiones, aclaro que nada impide que pongas el nombre que te dé la gana, pero por convención lo correcto debería ser lo que te he mencionado anteriormente. Por lo tanto, si quieres llevar esa librería a tu proyecto, quedaría con el siguiente aspecto:

1
2
3
4
5

[versions]
lifecycle-viewmodel = 2.8.1

[libraries]
lifecycle-viewmodel = { group = androidx-lifecycle, name = lifecycle-viewmodel, version-ref = lifecycle-viewmodel }

[plugins]

Por último en el documento libs.versions se encuentra plugins. Anterior a este sistema, las dependencias para los plugins se alojaban en el build.gradle del módulo del proyecto. Del mismo modo que las librerías, los plugins se han movido al libs.versions.

Las tablas que contiene plugins también se componen de pares clave-valor, en este caso id y version.ref.

El aspecto sería el siguiente:

1

android-application = { id = "com.android.application", version.ref = "agp" }

Del mismo modo que las librerías, la versión se gestiona en el bloque versions.

Comunicar libs.versions con build.gradle

Como he terminado con en el bloque de plugins, explicaré como se traslada esto al build.gradle del módulo de proyecto.

Antiguamente, el aspecto de este módulo era el siguiente:

1
2
3
4
5

plugins {
    id 'com.android.application' version '8.1.0' apply false
    id 'com.android.library' version '8.1.0' apply false
    id 'org.jetbrains.kotlin.android' version '1.7.20' apply false
}

La forma actual de tratar las dependencias de plugins es mediante alias:

1
2
3
4

plugins {
    alias(libs.plugins.android.application) apply false
    alias(libs.plugins.jetbrains.kotlin.android) apply false
}

Estructura de una dependencia en build.gradle modulo de proyecto

************************** Estructura de alias **************************
libs. plugins. mi.libreria
documento libs.versions.toml bloque de plugins mi-libreria

Detalle importante, en el documento toml se declaran las dependencias separando los nombres con guiones, mi-libreria y se trasladan al módulo sustituyendo los guiones por puntos mi.libreria, como habrás podido observar.

Estructura de una dependencia en build.gradle modulo de app

Por otra parte, el build.gradle del módulo app sigue utilizando implementation, pero en lugar de contener la dependencia ahora contiene una referencia a libs.

Tomaré el ejemplo de dagger-hilt que he mostrado antes, para mostrar como se vería en el build.gradle del módulo app.

1
2
3
4
5

dependencies {

    implementation(libs.libraries.hilt.android)

}

Conclusiones

Las ventajas de este sistema, en mi opinión son:

  • La unificación de todas las dependencias en un solo documento.
  • La claridad y limpieza que aporta a ficheros como el build.gradle del módulo app.
  • Estructura prácticamente común en ambos módulos a la hora de añadir tanto plugins como librerías.

Las desventajas son:

  • Entra en escena un tercer documento, añadir libs.versions no elimina la labor de tener que referenciar todas y cada una de las dependencias tanto en el módulo de la app, como en el módulo del proyecto.
  • Adaptarse al nuevo sistema sin tener información de su funcionamiento puede ser complejo para nuevos usuarios.

Si tenías dudas como yo acerca de este tema, espero haberte ayudado, saludos.
